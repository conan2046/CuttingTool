# CuttingTool 项目制作标准

> 编码：UTF-8  
> 适用范围：`D:\CuttingTool` 全目录  
> 优先级：本文件是项目内最高级制作规范；子目录如有更具体的 `AGENTS.md`，仅在不与本文件冲突时补充执行。

## 一、项目定位

本项目用于建立一套可复用的游戏 UI 资源生产流水线：

```text
文字需求/参考图
→ GPT Image 2 生成 UI 风格基准图
→ 按资源分类生成专用 Sheet
→ 本地确定性去背景、检测、切割、归一化
→ 人工或自动 QA
→ 导出透明 PNG、资源清单和预览总表
```

目标 Skill 名称：

```text
game-ui-asset-pipeline
```

该 Skill 负责工作流编排、提示词构建、图片后处理和验收，不等同于桌面 GUI 软件。桌面可视化编辑器属于后续独立产品层。

## 二、V1 交付范围

V1 必须支持：

- 根据文字和可选参考图生成 UI 风格基准图。
- 按分类生成面板、按钮、导航图标、通用图标、道具、装备、技能、状态资源和简单特效 Sheet。
- 为每个 Sheet 生成布局引导图。
- 使用纯色色键背景生成可拆分资源。
- 自动检测独立元素并生成包围框。
- 去除背景、清理色键边缘、保留真实 Alpha。
- 自动裁除透明空白，并按配置补充安全留白。
- 等比例归一化到目标画布。
- 分类、排序、命名并导出独立 PNG。
- 输出 `manifest.json`、`contact-sheet.png` 和 `qa-report.json`。
- 保留从生成源图到最终文件的可追溯关系。
- 同一请求包含多个分类时自动建立独立 Job；单分类超出容量时自动拆分多张 Sheet。
- 在所有生成图就绪后，通过统一确定性 Runner 汇总正式 Manifest、QA 和运行摘要。
- 对没有 Layout Guide 的未知整图先执行背景诊断，生成候选 bbox、可编辑修正 JSON 和标注预览。
- 在不引入桌面 GUI 的前提下，用批准后的 bbox 修正文件完成透明切割和正式 QA。

V1 不包含：

- 从被遮挡的完整 UI 截图中恢复不可见底层资源。
- 把假棋盘格当作真实透明通道。
- 自动生成最终可交互 UI Prefab。
- Unity 自动导入和 Sprite Editor 自动配置。
- 自动九宫格边界推断。
- 复杂烟雾、玻璃、液体、柔光等原生半透明资源的完全自动抠图。
- 动态 UI、序列帧动画和视频特效。
- 桌面 GUI 编辑器。

未经用户明确扩大范围，不得把以上后续能力混入 V1。

## 三、核心生产原则

### 1. 预览图和资源图分离

必须区分：

- `Canonical UI Style Reference`：用于确认整体风格，不用于正式切割。
- `Production Asset Sheet`：用于自动拆分，不承载界面预览、说明文字或排版演示。

禁止要求模型在同一张图中同时交付完整界面截图和可生产切片资源。

### 2. AI 负责视觉，脚本负责确定性

GPT Image 2 只负责：

- 风格设计。
- 元素造型。
- 材质、色彩和视觉品质。
- 按布局引导图生成相互分离的资源。

本地脚本必须负责：

- 精确行列和槽位定义。
- 色键移除。
- 连通域或槽位检测。
- 包围框计算。
- 透明边裁切。
- 尺寸归一化。
- 文件命名。
- Manifest 生成。
- QA 和错误报告。

不得依赖模型生成精确像素边界、精确切片坐标或可靠文件命名。

### 3. 风格基准优先

正式资源任务必须使用已批准的风格基准图作为视觉身份参考。每个分类 Sheet 应尽量同时附带：

1. `canonical-ui-style.png`：锁定风格、材质、描边、色彩和光照。
2. `layout-guide.png`：只规定槽位数量、间距、居中和安全区。

布局引导图是不可见施工参考，生成结果不得复制其中的网格、文字、中心点或辅助色。

## 五、Skill 制作标准

### 1. 安装与源码分离

源码位置：

```text
D:\CuttingTool\skills\game-ui-asset-pipeline
```

安装位置：

```text
C:\Users\Admin\.codex\skills\game-ui-asset-pipeline
```

始终先修改项目源码，再同步安装副本。不得只修改安装目录而不回写项目源码。

### 2. Skill 基本结构

`SKILL.md` 必须包含有效 YAML frontmatter：

```yaml
---
name: game-ui-asset-pipeline
description: <清晰说明触发时机、能力和输出>
---
```

要求：

- 目录名与 `name` 完全一致。
- `description` 覆盖用户可能使用的中文和英文表达。
- 正文使用祈使句，说明执行顺序和判断理由。
- `SKILL.md` 尽量控制在 500 行以内。
- 详细协议拆到 `references/`，并在正文中写明何时读取。
- 重复、确定性的任务必须进入 `scripts/`，不得每次临时重写。

### 3. 触发范围

应触发的典型表达：

- 生成游戏 UI。
- 制作 UI 资源包。
- 生成按钮、面板、图标、技能图标或道具图标。
- 生成可切割 UI Sheet。
- 把整张 UI 资源图切成透明 PNG。
- 去除 UI 素材背景并批量导出。
- 为 Unity 准备游戏 UI 图片资源。

不应因普通截图裁剪、照片抠图、角色动画或地图切片而误触发。

## 七、GPT Image 2 使用流程

### 1. 默认模式

正常图片生成使用内置 `image_gen`。除非用户明确要求 API/CLI，或确认需要原生透明回退，不直接调用 Image API 或自写生成脚本。

GPT Image 2 不提供原生透明背景时，默认使用：

```text
纯色色键生成 → 本地色键移除 → Alpha净边 → QA
```

不得谎称模型已经生成真实透明通道。

### 2. 色键选择

默认优先级：

| 资源内容 | 色键 |
|---|---|
| 不包含绿色 | `#00FF00` |
| 包含绿色 | `#FF00FF` |
| 同时包含绿色和品红 | 选择青色等冲突最小颜色，或拆分任务 |

选择色键前必须检查参考图和资源描述的主色。资源本体、描边、高光和阴影中不得使用色键及其近似色。

### 3. Sheet 强制约束

每个生成提示词必须明确：

- 精确资源数量。
- 精确行列布局。
- 从左到右、从上到下的资源顺序。
- 每个不可见槽位只有一个完整元素。
- 元素之间保留纯背景隔离。
- 元素不得接触画布边缘。
- 元素不得重叠、遮挡或跨槽位。
- 不生成文字、编号、标签、Logo 或水印。
- 不生成棋盘格、可见网格、分隔线和辅助线。
- 不生成地面、场景、投影或倒影。
- Layout Guide 只用于布局，不得复制到结果。

### 4. 推荐 Sheet 规格

| 类型 | 默认画布 | 默认布局 |
|---|---:|---:|
| Panel | `2048×2048` | `2×2` |
| Button | `2048×2048` | `3×4` |
| Icon_Nav | `2048×2048` | `4×4` |
| Icon_General | `2048×2048` | `4×4` |
| Icon_Item | `2048×2048` | `4×4` |
| Icon_Equip | `2048×2048` | `4×4` |
| Icon_Skill | `2048×2048` | `4×4` |
| Status | `2048×2048` | `3×3` 或按需求 |
| Effect | `2048×2048` | `4×3`，复杂效果应拆小批次 |

数量超过推荐容量时拆成多张 Sheet，不得为了塞入更多元素而牺牲单体清晰度。

批量拆分时，同一 `semantic_name` 的按钮状态或资源变体必须保留在同一张 Sheet。单个状态组超过整张 Sheet 容量时必须失败并要求调整布局，不得拆散状态组。

分类前缀：

```text
01_Panel
02_Button
03_Icon_Nav
04_Icon_Status
05_Icon_General
06_Icon_Item
07_Icon_Equip
08_Icon_Skill
09_Icon_Effect
```

文件名格式：

```text
<Category>_<SemanticName>_<StateOrVariant>_<Index>.png
```

示例：

```text
02_Button_Confirm_Normal_001.png
02_Button_Confirm_Pressed_001.png
06_Icon_Item_HealthPotion_001.png
08_Icon_Skill_FireSlash_001.png
```

规则：

- 英文文件名使用 PascalCase 语义名。
- 状态使用 `Normal`、`Pressed`、`Selected`、`Disabled` 等固定词。
- 编号使用三位数并在分类内连续。
- 不使用空格、中文、括号或临时描述。
- 不覆盖同名文件，除非用户明确要求替换。
- 默认生成稳定 ID，不使用随机 UUID 作为用户可见文件名。

## 九、切割和透明处理标准

### 1. 检测优先级

按以下顺序选择检测模式：

1. 已有 Alpha：Alpha 连通域。
2. 纯色色键：色差阈值和边缘洪水填充。
3. 已知网格：槽位内组件检测。
4. 自动检测失败：输出待人工校正的包围框，不擅自丢弃资源。

未知整图必须分类为 `alpha-sheet`、`flat-background-sheet`、`checkerboard-presentation` 或 `opaque-mixed-image`。只有前两类允许进入 bbox 修正导出；假棋盘格和未解析背景必须返回失败并说明需要重新生成或提供真实 Alpha。

候选 bbox 首次输出时 `approved` 必须为 `false`。Codex 查看标注预览并确认数量、边界、语义、分类、状态、启用项和连续编号后，才能改为 `true`。修正文件是当前无 GUI 工作流的正式人工介入点。

### 2. 连通域处理

必须支持：

- 排除小面积噪点。
- 合并属于同一资源的相邻碎片。
- 保留面板和图标内部的透明孔洞。
- 允许紧贴主体的硬边光效随主体合并。
- 禁止把两个槽位的资源合成一个元素。
- 禁止因内部镂空而把一个边框拆成四条边。
- 已知布局的槽位中心用于稳定归属和排序，不得把 Layout Guide 的槽位矩形直接当作硬裁切边界；允许完整主体伸入纯背景槽间留白，但主体合并、空槽和画布边缘接触仍判定为失败。

检测顺序必须稳定：从上到下、从左到右。

### 3. Alpha 和净边

输出必须为 RGBA PNG。验证：

- 四角或预期背景区域为透明。
- 完全透明像素的隐藏 RGB 被清理。
- 无明显绿边、品红边或色键污染。
- QA 统计可见色键残留时使用最低可见 Alpha 阈值；缩放插值产生的 `alpha < 16` 亚可见像素不判残色，`alpha >= 16` 的近色键像素仍判失败。
- 不截断描边、尖角和允许保留的紧贴光效。
- 不生成意外的内部透明条带或孔洞。
- 不使用多次反复羽化破坏清晰边缘。

每次流水线只安排一个明确的最终净边阶段，避免重复处理导致轮廓缩水。

图片模型生成的纯色色键可能存在轻微逐像素色差。若转换后所有槽位同时误报边缘接触或出现大面积背景连通域，应先统计边框色差分布，在确认主体不含近似色键后成对提高透明阈值和不透明阈值并重新验证；不得直接忽略边缘失败。

### 4. 归一化

归一化必须：

- 保持宽高比。
- 避免重复缩放。
- 根据分类选择居中或底部对齐。
- 保留配置的透明留白。
- 不放大低分辨率资源超过质量阈值。
- 记录原始包围框、缩放比例、对齐方式和最终尺寸。

## 十、Manifest 标准

每个导出资源至少记录：

```json
{
  "id": "06_Icon_Item_HealthPotion_001",
  "category": "Icon_Item",
  "semantic_name": "HealthPotion",
  "state": "Default",
  "source_sheet": "item-sheet-01.png",
  "source_index": 1,
  "source_bbox": [80, 96, 440, 460],
  "output": "Icon_Item/06_Icon_Item_HealthPotion_001.png",
  "width": 128,
  "height": 128,
  "padding": 8,
  "alignment": "center",
  "pivot": [0.5, 0.5],
  "qa": "pass"
}
```

Manifest 必须使用 UTF-8、稳定字段顺序和相对路径。不得保存不可移植的临时绝对路径作为唯一定位信息。

正式总 Manifest 的 `source_sheet` 和 `output` 必须相对于本次运行目录，例如 `generated/button-sheet-01.png` 和 `final/Button/02_Button_Confirm_Normal_001.png`。多 Sheet 分类使用分类内连续的 `category_index` 生成文件编号，不能在每张 Sheet 重新从 001 开始。

## 十一、运行目录标准

每次任务建立独立运行目录：

```text
output/<project-id>-<timestamp>/
├─ request.json
├─ jobs.json
├─ requests/
│  ├─ button-sheet-01.json
│  └─ icon-item-sheet-01.json
├─ references/
│  ├─ canonical-ui-style.png
│  └─ layout-guides/
├─ prompts/
├─ generated/
├─ extracted/
├─ normalized/
├─ final/
│  ├─ Panel/
│  ├─ Button/
│  ├─ Icon_Nav/
│  ├─ Icon_General/
│  ├─ Icon_Item/
│  ├─ Icon_Equip/
│  ├─ Icon_Skill/
│  └─ Icon_Effect/
└─ qa/
   ├─ contact-sheet.png
   ├─ qa-report.json
   └─ run-summary.json
```

多分类或多 Sheet 任务先运行 `prepare_ui_batch.py` 生成 Job、独立请求、Prompt 和 Layout Guide。所有 `generated_output` 就绪后运行 `run_ui_pipeline.py --run-dir <run-dir>`；Runner 负责原生 Alpha/色键分流、单次生产尺寸归一、逐 Job 切割、分类连续编号、总 Manifest、Contact Sheet、严格 QA 和 `run-summary.json`。Runner 不调用图片 API，缺少任何生成 Sheet 时必须在写入正式输出前失败。

未知整图先运行 `diagnose_ui_sheet.py`，输出 `input-diagnosis.json`、`bbox-corrections.json` 和 `bbox-preview.png`。修正确认后运行 `apply_bbox_corrections.py`；脚本必须拒绝未批准修正、假棋盘格、未解析背景、bbox 越界、bbox 重叠、bbox 切断前景和非连续编号。

只有在输出已经复制到项目运行目录、完成验证后，才能清理生成目录中的临时副本。

## 十二、QA 验收标准

### 1. 自动检查

必须检查：

- 输入图片可读取。
- 实际资源数量与请求数量一致。
- 每个槽位最多一个主要资源。
- 无空白资源。
- 无明显跨槽位重叠。
- 无画布边缘裁切。
- 输出具有 Alpha 通道。
- 背景透明度符合要求。
- 色键残留低于阈值。
- 透明区域不存在异常隐藏色。
- 文件名唯一且编号连续。
- Manifest 文件和实际文件一一对应。
- Contact Sheet 可生成。

### 2. 视觉检查

Contact Sheet 必须检查：

- 同一分类风格一致。
- 元素数量正确。
- 资源轮廓完整。
- 边缘无明显残色。
- 无网格、标签、文字或水印污染。
- 无错误合并或错误拆分。
- 同状态按钮轮廓和尺寸一致。
- 图标缩小到目标尺寸仍可辨认。
- 面板中央内容区干净，适合承载运行时内容。

### 3. 结果等级

```text
pass     可以交付
warning  可交付但必须记录问题
fail     禁止进入正式输出
```

不得把存在 `fail` 的资源包报告为完成。

## 十三、工具和依赖

### 1. Python

图片脚本优先使用 Codex 工作区捆绑 Python。运行前调用工作区依赖定位工具，并使用其返回的精确 Python 路径。

依赖至少包括：

- Pillow
- NumPy

当前 V1 核心流水线只依赖 Pillow 和 NumPy。工作区捆绑 Python 当前不包含 OpenCV，因此不得默认导入 `cv2`。只有现有 Pillow/NumPy 方案无法满足明确需求，并完成依赖、体积和可移植性评估后，才能引入 OpenCV。

不得假设裸 `python` 一定指向正确环境。

### 2. 文件操作

- 查找文件优先使用 `rg --files`。
- 查找文本优先使用 `rg`。
- 小规模文件修改使用 `apply_patch`。
- 禁止用脚本重写与任务无关的文件。
- 中文 Markdown、JSON 和配置使用 UTF-8。
- 删除或移动前确认目标绝对路径位于项目目录或明确授权目录内。

### 3. 日志

脚本日志必须：

- 包含时间、阶段、输入、输出、结果和错误原因。
- 不输出 API Key、认证信息或用户隐私。
- 错误信息可直接定位到文件和资源编号。
- 自动化脚本失败时返回非零退出码。

## 十四、代码标准

- Python 使用类型提示。
- 路径统一使用 `pathlib.Path`。
- JSON 使用 UTF-8，写出时 `ensure_ascii=False`。
- 每个脚本提供清晰 CLI 帮助。
- 不硬编码个人临时目录。
- 图像尺寸、阈值、色键和留白必须参数化。
- 算法输出尽量确定性；同输入、同配置应生成同结果。
- 自动排序不得依赖文件系统未定义顺序。
- 不在确定性脚本中程序化绘制最终美术资源。
- 脚本只处理、切割、验证和组装 AI 或用户提供的资源。

## 十五、测试标准

### 1. 测试运行器

- 本项目默认且唯一规定的测试入口是 Python 标准库 `unittest`：

  ```powershell
  & $PYTHON -m unittest discover `
    -s .\skills\game-ui-asset-pipeline\tests `
    -p 'test_*.py' `
    -v
  ```

- 不得先尝试 `pytest`、不得因通用习惯探测或安装 `pytest`，也不得把缺少 `pytest` 报告为项目依赖问题。
- 只有用户明确批准测试框架迁移，并同步更新 `pyproject.toml`、README、测试代码和本文件后，才允许改用或新增 `pytest`。
- 文档或资源生产任务如未修改代码，可按风险决定是否重跑测试；修改核心脚本时必须运行与改动范围匹配的 `unittest` 测试。

### 2. 覆盖要求

每个核心脚本至少覆盖：

- 正常输入。
- 空图或损坏图。
- 数量不足。
- 数量超出。
- 元素接触边缘。
- 两个元素意外相连。
- 内部透明孔洞。
- 小碎片噪点。
- 色键接近主体颜色。
- 金色、白色和深色抗锯齿边缘。
- 文件重名。
- Manifest 与文件不一致。

至少保留三类端到端样本：

1. 标准纯色色键 Sheet。
2. 本项目附件一类的假棋盘格/混合展示图，用于验证正确拒绝或降级处理。
3. 包含重叠、裁边、残色的失败样本。

未知整图样本必须由自动测试直接读取，不能只在临时目录中动态构造后删除。标注预览和正式 Contact Sheet 都必须完成视觉检查。

批量编排还必须覆盖：至少两个分类、至少一个分类跨两张 Sheet、同一按钮状态组不跨 Sheet、色键与原生 Alpha 输入混合、跨 Sheet 分类编号连续、缺少生成图时预检失败。

完成修改后，必须运行与改动范围相匹配的测试。不得只以“脚本执行成功”代替视觉和数据验收。

## 十六、开发流程

执行顺序固定为：

1. 读取本文件和目标 Skill 的 `SKILL.md`。
2. 检查工作区状态和已有用户修改。
3. 明确本次任务属于协议、提示词、生成、切割、QA 或导出中的哪一层。
4. 只修改必要文件。
5. 先做确定性单元测试。
6. 再做小样本端到端测试。
7. 生成 Contact Sheet 并视觉检查。
8. 更新说明和测试记录。
9. 同步安装副本。
10. 在新任务中验证 Skill 触发。

不允许跳过源码与安装副本一致性检查。

## 十七、完成定义

任务只有同时满足以下条件才算完成：

## 十八、重大变更边界

## 十九、AGENTS.md 同步维护规则

本文件必须与工具当前能力保持一致，不得只更新代码而遗留过期制作标准。

每次发生以下变化时，同一任务内同步检查并更新本文件：

- 新增功能、资源类别、输出格式、脚本、依赖或工作流阶段。
- 优化生成提示词、切割算法、色键处理、归一化、命名或 QA 标准。
- 修复 Bug，且修复揭示了新的边界、失败模式、禁止做法或验证要求。
- 修改目录结构、文件职责、运行命令、安装路径或交付方式。
- 改变 GPT Image、Python、Unity 或其他外部工具的使用策略。
- 增加用户必须知道的限制、兼容性要求或人工介入条件。

同步要求：

1. 功能代码、测试、使用文档和本文件在同一任务内完成更新。
2. 在最相关章节修改规范，不把所有变更堆成无上下文的流水账。
3. 新增规则要说明适用范围、原因、执行方法和验收方式。
4. Bug 修复至少补充对应测试；若属于通用失败模式，同时补充边界或 QA 规则。
5. 如果改动不影响制作标准，在交付说明中明确记录“已检查 AGENTS.md，无需修改”。
6. 完成任务前，将“AGENTS.md 已同步或已确认无需修改”纳入完成检查。

禁止出现以下情况：

- 代码已经支持新能力，但本文件仍声明不支持。
- 命令、目录或依赖已改变，本文件仍保留旧方法。
- 同类 Bug 已修复，但本文件和测试没有记录预防规则。
- 为了记录变更而删除仍然有效的历史边界或制作标准。
